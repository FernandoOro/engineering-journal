## What I did

### 1. Crash fix that was killing the flow
- **Symptom:** an `AttributeError` deep in a memory component took down the **entire** flow; it
  surfaced as a **client timeout** because the flow died without responding.
- **Fix:** the agent resolver now **filters out non-agent specs** before resolving, so a non-agent
  spec can't blow up resolution.

- Added observability for **per-agent and per-model latency** (to see who eats the time), and on the

## Learnings / technical decisions
- **"Timeout" is often a *crash* in disguise.** The client saw a timeout, but the flow was dying on an
  `AttributeError`: it wasn't slow, it was dead. Before optimizing latency, confirm the flow **isn't
  dying**.
- **Instrument before optimizing.** I didn't touch the swarm until I had **per-agent/per-model
  latency**; the logs pointed at the serial sub-phase as the real cost. Parallelizing blind would
  have been guessing.
- **SSE needs a heartbeat on long jobs.** A multi-agent flow that takes minutes needs an
  **idle-heartbeat** so the client doesn't drop the connection, plus a **total cap** as a safety net.
- **A per-agent cap = no single agent hangs the whole flow.** An `execute()` with no per-agent ceiling
  lets one slow agent hijack the entire swarm.
- **Testing "with every prompt" isn't optional in a multi-agent system.** Each prompt exercises a
  different combination of agents; the memory-component crash only showed up with certain specs.
  Exhaustive testing is what buys the confidence — and it's why it **takes the time it takes**.

## What I could do better
- Manual testing across every prompt is valuable but **slow and non-cumulative**: I repeat it each
  release. I should capture those prompts as a **regression set** (even a smoke set) so I don't re-run
  the whole catalog every Friday.
- I hit the credit limit with the **seats** piece half-started. Same pattern as Monday: reserve the
  big tasks for the **start** of the session and leave the closeable ones for the end.
