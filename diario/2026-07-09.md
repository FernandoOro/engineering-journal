## What I did

### 1. "Primary provider never serves" alert (Observer pattern, end-to-end)
- Spec: evaluated three options — an Observer on top of the existing provider-health tracker (A),
  an explicit state machine (B), a polling background job (C). Chose **A**: reuses the existing
  tracker, one hook point for all three call sites of the carousel. B was dropped for solving a
  requirement nobody asked for (no need for intermediate states); C for adding infrastructure A
  doesn't need.
- Implemented consecutive-failure counting per provider (alert threshold N=5), wired the **actual
  outbound alert call** (previously a stub with no real effect), and a recovery notification the
  first time the primary provider serves again. The implementation hooked 2 of the 3 planned
  carousel call sites; the third stayed a documented gap — closed hours later, the same day, as a
  side effect of the next item.

### 2. Carousel per-attempt timeout (first half of a two-part ticket)
- Spec: reused an existing per-agent timeout value (already calibrated per agent type, 20-45s)
  instead of inventing a new global constant.
- Wrapped each carousel attempt in a bounded wait, falling through to the next model if an attempt
  exceeds its budget. Directly mitigates an incident where a single slow-but-not-erroring attempt
  had been consuming the client's entire timeout budget. Explicitly out of scope: a legacy,
  synchronous version of the same carousel — the real incident runs on the current version, not
  the legacy one; documented as debt if the legacy path ever needs the same treatment.
- Same change: since this ticket already touched the exception-handling path of the third call
  site, used the opportunity to close the alerting gap from item 1 right there too — same pattern
  as the other two call sites.

### 4. SDK rewrite — three slices, complete

  Deleted it (a few hundred lines, no tests to lose) instead of keeping it around "just in case."

## Learnings / technical decisions
- **A half-finished stub is worse than an empty one.** The alert function already had the right
  shape (channel + params) but never made the real call — nobody would have noticed until the
  alert "worked" in the logs but never reached anyone. Verify a function's *effect*, not just its
  signature.
- **Before repurposing code "in case it's needed later," find the real consumer.** A dead consumer
  class had sat unused for months "in case" another surface needed it; a whole-monorepo search
  took minutes and avoided carrying a few hundred lines of speculative debt.
- **Correcting an in-flight ticket's scope, with it documented, beats implementing something with
  no consumer just because it was in the original ticket.** The dropped parameter in slice 2 is the
  example — verify against the real code before building.
