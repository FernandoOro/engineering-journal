## What I did

### 1. Fix an orphaned periodic-sync task queue (ship-first)
- **Problem:** 8 scheduled (beat) entries pin a queue name that **was never defined**, and routing
  sent those tasks to `default`. Result: the tasks landed in a **queue no worker consumed**, so the
  periodic sync engine **never ran** (campaign syncs, rate-limit resets, cleanup, reconciles…).
- **Decision (isolate):** defined a **dedicated queue + a worker** that consumes it, rather than
  folding it into `default`. Isolates the sync from `default` contention. Added the dedicated worker
  to the dev compose.
- **Spec:** documented the A/B choice and rationale.
- **Validation:** beat + worker up; a scheduled task is received and completes; the sync-log table
  gets a fresh row after a cycle.

### 2. Per-user/day run-rate gate on the multi-agent product
- **Problem:** the streaming endpoint validated only auth + a feature flag before dispatch → an
  authenticated user had **unlimited runs**, and each run fans out to several LLM+MCP agents (direct
  cost and abuse surface).
- **Fix:** a rate-limit **gate before dispatch** — a Redis counter per user/day (`INCR` +
  end-of-day `EXPIRE`, atomic), a **typed 429** on exceed, and a **fail-open feature flag** so it can
  be disabled without redeploy. Reused the one quota pattern an audit flagged as working.
- **Docs:** added the rate-limit tables to the persistence section.

### 3. Surface usage at the LLM boundary (★ root unlock)
- **Problem:** the LLM boundary functions returned a **bare string**, discarding the provider's
  `usage` (prompt/completion tokens, model). Without it there's **no cost, no budgets, no cost-based
  limits, no per-agent metrics** anywhere in the product. This is the root blocker.
- **Fix:** both boundary functions now return **text + a usage struct**
  `{prompt_tokens, completion_tokens, total_tokens, model, provider}` (null-safe when a provider omits
  it), with a **compatibility shim** for callers that only read text. No change to
  selection/fallback logic — usage capture only.

### 4. Token/cost fields on the agent result (PR 1)

## Learnings / technical decisions
- **A queue with no worker is a cron that never runs.** Same pattern as last week's orphaned queue: a
  scheduler pointing at a queue with no consumer = the sync **never** executes, silently. The rule is
  **one consistent choice**: either define the queue and have a worker consume it, or route to a queue
  that's already consumed — never both, half-done.
- **Capturing usage at the boundary instruments everything at once.** This is a Template-Method seam:
  turn the bare string into `text + usage` at the single boundary point, and every agent is
  instrumented once the result type carries it — **without touching each agent**. Finding the single
  correct point is what makes the change small instead of huge.
- **A rate limiter that isn't atomic isn't a rate limiter.** The counter uses `INCR` + `EXPIRE`; if it
  weren't atomic, two simultaneous requests at the boundary both pass. And **fail-open behind a flag**:
  a new gate you can turn off without a redeploy can't become the incident itself.
- **One runner per environment.** Sharing a self-hosted runner between dev and prod couples deploys
  that must be independent; **dedicated labels** are the minimal correct separation.
- **Spec-driven: Done only with evidence.** Writing the spec (contract + plan + validation) *before*
  coding forces you to define acceptance first; status isn't "Done" until you paste the real evidence.
