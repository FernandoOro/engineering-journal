### 2. Local Dockerization (`docker-compose.local.yml`) + design & security decisions
Goal: a local stack that brings up the API, Celery/Flower, Postgres, and Redis reproducibly,
aligned with production but with no hardcoded credentials.

#### 2.1 Postgres with semantic/vector DB support — not the base image
- Switched the Postgres image from the base `postgres:16` to **`pgvector/pgvector:pg16`**,
  because the semantic-memory tables need the **`vector` extension**, which the base Postgres
  image doesn't ship.

#### 2.2 Fixing the Flower container
- **Cause:** Flower (the Celery monitoring UI) wasn't installing inside the container because
  it was declared as a **dev dependency**, and the Docker image only installs main
  dependencies.
- **Fix:** moved `flower` into main dependencies so it installs in the image.
- Also added **healthchecks** to the `api` and `flower` services so Compose knows when they're
  actually ready, not just "started".

#### 2.3 Removing hardcoded DB credentials (security fix)
- **Problem:** the database user/password/name were **hardcoded** in the compose file (the
  Postgres service's `environment:` block).
- **Fix (two steps):**
  1. Align the credentials the service used with the environment file (they didn't match).
  2. Replace the literal `environment:` block with **`env_file`**, so the Postgres container
     reads its credentials directly from the env file instead of having them committed in the
     versioned YAML.

### 3. GitHub Actions research for secrets (local / dev / prod)
- **Motivation:** stop shuttling `.env` files between environments by hand. Each `.env` is a
  place a value can leak. The plan is to keep **secrets in GitHub Actions Secrets** and let the
  pipeline inject them at deploy time.
- **Concrete progress:** added a first workflow that, on `push` to the `dev` branch, writes the
  env file and a cloud service-account credential from repository **secrets**, brings up the
  stack with Docker Compose, and polls a health endpoint until the API responds.
- **Implementation finding:** deploying to a **local** environment via Actions requires a
  **self-hosted runner** (`runs-on: self-hosted`), because GitHub-hosted runners can't reach
  the local machine.
- **Blocker:** all three flows (local/dev/prod) need **tokens/secrets** that only the backend
  repo admins hold. Waiting on those tokens to wire up dev and prod.

## Learnings / technical decisions
- **The base image isn't always enough.** A semantic/vector DB needs the `vector` extension,
  so `pgvector/pgvector:pg16` rather than `postgres:16`. The container image is a capability
  decision, not a detail.
- **"Works on my machine" ≠ "works in the container".** Flower didn't install because it was a
  dev dependency and the image only carries main ones. Packaging boundaries (dev vs prod deps)
  matter as much as the code.
- **Credentials out of the versioned YAML.** `env_file` instead of literal `environment:`
  values keeps the DB user/password from being committed. Same principle throughout: a secret
  never lives in a file that goes to the repo.
- **Deploying locally via Actions needs a self-hosted runner.** Hosted runners can't see your
  machine; the self-hosted one runs in the target environment.
- **A healthcheck ≠ "the container started".** A live process doesn't mean a ready service; the
  healthcheck models "ready to serve", and Compose can order startup on it.

## What I could do better
- The credential misalignment (one commit to align, another to remove) came from not defining a
  **single source of truth** for the env variables up front. Pointing compose at `env_file`
  from the start would have saved the intermediate "align then remove" step.
- I'm blocked on **tokens** (only admins hold them). I could have anticipated this and
  requested them at the start of the day so the dev/prod wiring wasn't stalled. Logged
  explicitly to unblock tomorrow.
