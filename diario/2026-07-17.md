# 2026-07-17 (Friday)

**Context:** Reorganized SDK calls by business domain · closed out a billing feature (create a plan
with an optional project association) · five UI fixes on a billing modal · fixed a project-switcher
bug · a batch of specs closing out the week's documentation.

Closed out the week on three fronts: in the morning, re-organized SDK calls that had ended up in
the wrong module (same principle applied Monday to the chat client SDK, now applied at the level of
two other SDK modules); in the afternoon, closed out a billing surface (plan creation with an
optional project link) with several UI fixes on its modal, including a real bug in the project
switcher that had slipped in since Thursday; and at the end of the day, documented in one sitting
the nine specs still missing from the week.

## What I did

### 1. Reorganized SDK calls by domain (morning)
- Moved a couple of project-listing calls into the module matching their actual business domain.
- Moved an overview-data call and team/member calls into a different module, for the same reason.
- Same principle as Monday's chat-client SDK: each call lives where its business domain actually
  belongs, not where it was first written.

### 2. Billing — plan creation now accepts an optional project name
- Creating a plan can now associate it with a project by name in the same call.
- Closed out two backend branches that had been open since Wednesday/Thursday.

### 3. A project-creation modal wired to plan creation, plus five billing UI fixes
- Wired the modal to the plan-creation endpoint (billing cycle, coupon, and project name).
- Show the coupon's discount amount after applying it.
- Hide the billing-cycle toggle for fixed-price plans and show the already-discounted price.
- The top-tier plan hides the coupon field and changes its submit button to "Contact us."
- Properly separated two fields (billing cycle and coupon) that had been unintentionally linked —
  changing one affected the other.

### 4. Fixed a bug — the project switcher wasn't actually switching the active project
- Selecting a project in the switcher looked like it worked (visual change) but didn't actually
  change the active project. A bug that had slipped in since the switcher was wired to live data on
  Thursday; found and fixed today.

### 5. Documentation close-out — nine specs in one sitting
- Wrote the remaining specs for the week's features: a plan-creation flow, moving some data calls
  between SDK modules, wiring live data to a few more screens, and three backend endpoints from
  earlier in the week.

## Ops
- Four merges closing out the week's branches.
- Nine spec documents written in a short window — a documentation close-out, not spec-first.

## Learnings / technical decisions
- Organizing SDK calls by business domain instead of by feature makes them easier to find later by
  what they do, not by when they were written.
- Explicitly separating two fields that had been entangled prevents a fix to one from breaking the
  other — the root cause behind at least two of today's UI fixes.

## What I could do better
- The project-switcher bug is exactly the kind of thing a manual click would have caught on
  Thursday, before marking that feature done — it went unnoticed for a full day until I found it
  today.
- Today's nine specs are retroactive documentation of features already implemented on earlier days
  (same pattern noted yesterday) — writing the spec the same day as the implementation, rather than
  in a batch at the end of the week, would be more faithful to the process we're supposed to
  follow.
