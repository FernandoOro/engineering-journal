### 5. Security improvements for the third-party integration — part 1
A sweep over the integration's routers, documenting each finding in a tracked security catalog
and fixing them. The catalog grew from **24 → 28** findings.

- **Defensive auth on routers that had none:**
  - One router exposed ~25 endpoints (campaigns, analytics, webhooks) **with no router-level
    authentication** despite handling CRM data. Fix: add the auth dependency to the **root
    `APIRouter`** so the JWT check propagates across the whole tree **without touching the
    handlers**.
  - A monitoring router (cancel/retry/workers/queues — infrastructure operations) was only
    behind "authenticated user". I **raised it to an admin-role requirement** at router level;
    it must not be reachable by regular users.
  - Pattern: router-level `dependencies=[Depends(...)]` — defensive by default; sub-routers
    inherit the check with no per-handler duplication. (CWE-306 / CWE-284.)

- **Broken OAuth callback + an unmounted admin router:**
  - The OAuth callback route **always ran a legacy stub**, shadowing the real callback. Removed
    the legacy GET handler so the real one is the only handler. **Verified live:** a dummy code
    now returns the provider's `invalid_code` (previously it returned the legacy "this is a
    legacy endpoint" response). The code→access+refresh token exchange works end-to-end.
  - An admin router was written and auth-guarded but **never mounted** in the app entrypoint
    (an oversight, not a decision) → 7 admin endpoints were unreachable. Added the
    `include_router()`; the admin-role guard was already on the router.
  - After this, the open-security list dropped to **2** items.

## Learnings / technical decisions
- **Router-level auth beats per-handler auth.** Putting `dependencies=[Depends(...)]` on the
  `APIRouter` protects the whole tree at once and won't be forgotten on a new endpoint. The bug
  was the opposite: trusting each handler to remember to ask for auth. Defensive by default.
- **Privilege by operation, not just "is logged in".** Cancelling/retrying background workers is
  infrastructure → require an admin role, not merely an authenticated user.
- **A legacy stub can shadow the real handler.** The OAuth callback "worked" in code, but the old
  GET route always intercepted it. Verifying **live** (dummy code → `invalid_code`) is what
  confirmed the correct handler now runs.
- **Code that exists but isn't mounted = doesn't exist.** The admin router was written and
  guarded, but without `include_router()` it was unreachable. Treat the app entrypoint as the
  inventory of what's actually exposed.
- **Named volume = persistence.** A container is ephemeral; data survives because the volume
  lives on the host. Redis also needs RDB/AOF enabled to flush to that volume.

## What I could do better
- The four findings (missing auth, shadowed OAuth, unmounted router) are exactly what a
  **provider-integration checklist** would catch up front: does the router have auth? is the real
  callback the one that runs? is it mounted in the entrypoint? Worth turning into a template for
  the next integration.
