# 2026-06-09 (Tuesday)

**Context:** Backend on a fintech SaaS platform (client) — authorization/identity
security + billing.

## What I did

### 1. IDOR via resource identifier
- **Diagnosis:** several endpoints exposed data keyed by a chat identifier without
  verifying the caller belonged to that chat.
- **Decision:** a reusable FastAPI dependency (**Proxy** pattern) that validates
  membership before exposing anything.
- **Implementation:** the dependency covers all three roles (owner, member, client) and
  protects four read endpoints, returning `401` without a token, `403` for someone
  else's resource, `404` for a non-existent one.
- **Tests:** 10/10 scenarios validated against the running backend.

### 2. Identity forgery via falsifiable request fields
- **Diagnosis:** some endpoints trusted a client-supplied session/user id as the active
  identity — forgeable by the caller.
- **Implementation:** identity now comes **exclusively** from the verified JWT. On upload,
  the session id is kept only as an optional tracing field. On the plan endpoints, the
  client-supplied user id is removed as identity; all internal uses (coupon service,
  no-charge activation, CRM notifier, payment metadata) switched to the JWT-derived id.
  Audit log lines intentionally retained.

### 3. Billing: upgrade sets seats and creates subscriptions
- **Implementation:** compute the unassigned-seat delta on plan migration
  (`old_unassigned + max(0, new_included − old_included)`); create form + seat
  subscription records for net-new included seats; add a fallback plan subscription when
  the payment provider hasn't provisioned one yet (allowing a `0`-amount internal record).
- **Tests:** integration covering the tier-upgrade delta and additive unassigned seats.

## Ops
- Manual checks of the protected endpoints with tokens from different roles to confirm
  the `401/403/404` codes; validated the upgrade against a local DB by checking the seat
  counts before and after.

## Learnings / technical decisions
- **Identity golden rule:** always derive identity from the verified session (JWT), never
  from a client parameter. A session id can stay for tracing, never for authorization.
- Centralizing access control in one dependency avoids repeating the check in every
  handler and removes the risk of forgetting it in a new one.

## What I could do better
- The identity fix took two commits because the first missed some client-supplied-id
  uses inside the upgrade path. An exhaustive `grep` for those fields *before* starting
  would have closed the hole in a single pass.
