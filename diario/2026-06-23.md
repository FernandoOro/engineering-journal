- **Deploy-flow learning (registry + CI secrets):**
  1. Create/use a container-registry account → a **Personal Access Token** with read+write.
  2. On the **server**, log in with the account and the PAT.

## Learnings / technical decisions
- **Killing the static token closes a whole class of risk.** As long as long-lived `*_ACCESS_TOKEN`s
  live in config, there's something to leak and to expire silently. With the manager + refresh you
  don't need to store them: the finding closes **by design**, not by patch.
- **One secret per API key, config in Variables.** Sensitive values go to CI *Secrets* (one per
  key); non-sensitive to *Variables*. Don't mix, and remember CI won't let you re-read a secret — if
  you doubt the value, rotate it.
- **`docker-compose.dev.yml` should mirror local.** If dev and local diverge in services or shape,
  "works locally" stops predicting "works in dev". Homogenizing the compose shape is the work, not
  cosmetics.
- **A security cleanup has a hotfix tail.** Removing the transient attribute broke a connection test;
  any cleanup touching shared state needs a "what else read this?" pass.

## What I could do better
- The connection-test hotfix was predictable: when removing an attribute, I should `grep` **all** its
  readers first, not after the test fails.
- I documented the registry+CI flow in my notes, but it should live as a **runbook in the repo** so
  the next person to deploy doesn't relearn it.
