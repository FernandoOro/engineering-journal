## Learnings / technical decisions
- **Prod exposes nothing; dev can.** The right network surface for production is **zero published
  ports** for DB/Redis/monitoring (reachable only inside the Docker network); dev can afford to open
  them for inspection. Mixing the two criteria is how data services leak.
- **One deploy file per environment.** Having two compose files for prod was a trap: two sources of
  truth. One file per environment, consistently named.
- **The fallback's limit isn't technical, it's configuration.** The provider code is tested; what's
  left (verify the domain/API key) is on management's side. Knowing **where my part ends** avoids
  chasing a problem that isn't in the code.
- **The full flow test is ~2h by design** (it waits for all reports to finish). Plan it as a long
  block, not a quick check.
- **Back up before Dockerizing the DB.** Moving real prod data into the Postgres container requires a
  prior backup; the dockerized-prod-with-realistic-data mode depends on that restore.

## What I could do better
- The dev/prod port asymmetry I fixed across several commits (add containers → remove exposure → fix
  Redis ports). A **network-hardening checklist** ("what gets published to the host in prod? no data
  services") would have gotten it right in one pass.
- The 2h flow test is valuable but **manual**; a **runbook** of what to observe at the end (all
  reports finish, containers healthy) would remove reliance on memory.
