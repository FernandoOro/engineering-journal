## What I did

### 1. Closing the recurring admin-auth gap (structural guardrail, not just the named route)
- Added the missing auth dependency to a metrics router (multiple sub-endpoints), plus a
  **generic regression test** that walks every admin/metrics router and fails CI if any of them
  loses the auth dependency — not just the one route named in the ticket.
- Building that test against the real code (not a surface-level grep) surfaced two routers with
  **partial** protection — worse than none, because it looks covered. Documented as explicit
  follow-up debt rather than papering over it: widening scope to 30+ routers would have turned a
  small ticket into a large one and delayed the rest of the team.
- Also closed out a stakeholder's earlier finding: a dangling middleware import was already gone
  from the app entrypoint (verified by direct search), and the class behind it was a no-op stub —
  corrected the docs to reflect that.

### 2. Authenticated campaign-edit endpoint, shipped with my own QA
- Added an authenticated update endpoint for a resource type, reusing the exact pattern from an
  existing status-update endpoint (read → merge a whitelist of fields → write back). All ~15
  editable fields land in a single pass (no migration — persisted as a flexible document), guarded
  by an **explicit whitelist** of editable fields — any key outside it is ignored even if the
  frontend sends it by mistake.
- **My own QA in the dev environment found two spec gaps before marking it done:** the read model
  silently dropped ~10 "extended" fields (they persisted but never came back on GET); and without
  a real ownership check (it was mocked, a pre-existing issue), any user could edit any resource by
  guessing an ID. Fixed both: declared the missing fields on the model, and added an ownership
  check that returns **404, not 403**, so an unauthorized caller can't learn that a resource exists
  under another tenant.
- Found and filed separately (non-blocking) four pre-existing bugs. Fixed two of them the same day
  (see #4).

### 3. Root-cause correction on a credentials incident — the secret was never wrong
- The previous week's diagnosis ("invalid cloud credentials") was **wrong**: the secret was always
  valid. The deploy workflows interpolated the secret **raw inside a shell string** — since the
  credential JSON contains double quotes, those quotes prematurely closed the shell string,
  corrupting the credential file on **every deploy**. Proof: injecting the same secret as an
  environment variable instead produced a valid file.
- Hotfixed the deploy workflow using the environment-variable pattern, did a hot prod redeploy to
  pick it up, then merged the same fix for the dev deploy workflow (same bug, dev writes the same
  prod credential on purpose).
- **Verified in production:** logs went from credential errors / 100% fallback to a secondary
  provider, to the primary model serving normally again.

### 4. Two pre-existing bugs fixed (surfaced by my own QA)
- A document type wasn't serializing a datetime field before persisting, breaking a create
  endpoint with a 500.
- An ownership-check function was **mocked** to always return true — no real authorization effect.
  Fixed it even though it wasn't blocking the campaign-edit ticket (already flagged as a QA finding
  in #2).

## Learnings / technical decisions
- **Re-verify your own prior diagnosis, not just new code.** I closed the credentials incident the
  week before as "invalid secret"; with more evidence (testing injection as an env var) it turned
  out the secret was always valid and the bug was in how CI wrote it. A "closed" diagnosis isn't
  untouchable if new evidence shows up.
- **Never interpolate a secret directly into a shell `run:` step — always via an environment
  variable.** Textual interpolation plus quote-containing content is a combination that breaks
  silently and doesn't fail CI loudly (the file gets written "successfully," just with corrupted
  content).
- **My own QA pass before marking something done caught two bugs no unit test would have** (a
  validation library silently dropping undeclared fields; a missing ownership check). Manual
  end-to-end verification still catches things the code itself doesn't anticipate.
- **Return 404, not 403, when an ownership mismatch could leak existence.** A real
  security-relevant API contract detail — don't confirm to an unauthorized caller that a resource
  exists under someone else's account.

## What I could do better
