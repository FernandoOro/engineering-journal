# 2026-06-08 (Monday)

**Context:** Backend on a fintech SaaS platform (client) — security hardening sprint, part 1.

## What I did

### 1. Secrets out of version control + correct ignore rules
- **Diagnosis:** ignore rules only covered the generic `.env`, leaving environment-
  specific variants and service-account/credential files tracked.
- **Implementation:** untracked those files from the index, expanded the ignore rules to
  cover every environment variant and credential file, and kept them on disk (needed by
  the container/build tooling) with a local gitignored backup. Also removed a dead
  middleware import that was never actually registered.

### 2. SSRF-safe web-scraping tool
- **Decision:** wrap the scraping tool using the **Decorator** pattern instead of editing
  it, so the security policy lives in one auditable place.
- **Implementation:** a safe wrapper validates target URLs against **RFC 1918**,
  link-local (`169.254.x.x` / cloud metadata), loopback, and private IPv6 ranges before
  delegating, and **fails closed** on DNS errors. Replaced the bare tool at its two call
  sites.
- **Tests:** 17 unit, all green.

### 3. Stop leaking exception detail in debug endpoints
- **Diagnosis:** debug endpoints returned `str(e)` in responses, exposing internal detail.
- **Implementation:** generic messages in both inner and outer except blocks; full detail
  logged server-side with stack info. Verified the global exception handler already
  returned a generic `{"detail": "Internal server error"}` without leaking message, file
  paths, or credentials.
- **Tests:** 4 unit on the global handler (run in <1s, no containers) plus endpoint tests.

## Ops
- Verified the app still booted with the secret files on disk (build tooling intact), and
  ran the security unit suites before pushing.

## Learnings / technical decisions
- The **Decorator** pattern for SSRF keeps the original tool untouched and concentrates
  the policy in one testable place.
- **Fail-closed on DNS error** is the right call: when in doubt, don't resolve and don't
  make the request, rather than risk reaching an internal resource.
- A generic `.env` ignore gives a false sense of safety — the suffixed variants are
  exactly the ones carrying real credentials.

## What I could do better
- I untracked the secrets, but they remain in git history. Rotating the potentially
  exposed credentials (and, where warranted, scrubbing history) is the necessary
  follow-up — logged as security debt to prioritize.
