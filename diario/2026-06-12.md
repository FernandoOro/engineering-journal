# 2026-06-12 (Friday)

**Context:** Backend on a fintech SaaS platform (client) — full SEO engine build +
documentation.

A big day: build the SEO engine end to end on the previous day's data layer, move blog
generation to an async streaming flow, fix the multi-tenancy model (two iterations), and
document everything.

## What I did

### 1. SEO engine — full implementation
- **Implementation:** a complete module with Search Console sync, AI blog writing, weekly
  ideas, and WordPress publishing — dashboard/connectors/reports/blogs/ideas routes, a
  service per concern, and Celery tasks (daily search-console sync, Monday idea
  generation) routed to a shared queue.
- **Fixes applied after code review (same commit):** made the tenant column nullable to
  match the dev DB; fixed a column that mistakenly held the wrong id; corrected the
  search-console dimension mapping; added per-period `rollback()` to isolate sync
  failures; fanned out the all-orgs sync per tenant and closed the session before the
  loop; corrected the beat queue; added not-found guards; wrapped the ideas task in a
  timeout with retry.
- **Tests:** 78 unit + 17 integration = **95/95** green.

### 2. Tenant-key refactor #1
- **Diagnosis:** the tables keyed on an organization id, but auth uses a JWT whose
  primary key is the user account — so tenant resolution always 404'd.
- **Implementation:** migrated models, services, routes, and Celery tasks to the user
  key, dropping the now-dead resolution helpers and passing the verified identity
  straight through. 95 tests updated and passing against a real DB.

### 3. Blog generation → async (Celery) + streaming (SSE)
- **Decision:** the generate endpoint must not block ~60s; it should dispatch a task and
  return a task id immediately.
- **Implementation:** a new SSE stream endpoint (Redis pub/sub) and a task that runs the
  writer, persists the result, and publishes a "done" event — same pattern as the ideas
  stream.

### 4. Blog detail endpoint + auth/ORM hardening
- **Implementation:** an endpoint to save direct markdown edits; strengthened the
  current-tenant dependency with membership validation; refactored ORM models with
  explicit fields and correct types; more robust error handling in the SSE and weekly task.

### 5. Tenant-key refactor #2
- **Diagnosis:** the previous change moved models and services to a finer-grained tenant
  key but the matching migration was **left uncommitted**, leaving the schema behind the
  code.
- **Implementation:** a migration over nine tables swapping the tenant column and
  rebuilding all indexes and unique constraints around the new key.

### 6. CORS for a custom header
- **Implementation:** allowed the frontend's tenant header through CORS so preflight stops
  rejecting requests that every SEO endpoint needs.

### 7. Documentation
- **Implementation:** documented the engine end to end (~25 files): frontend routes and
  api-client, backend endpoints, the async + streaming blog flow, the blog editor spec,
  and the tenant migration with its architecture changes.

## Ops
- Two `alembic upgrade` runs in the day (both tenant migrations), applied and verified;
  re-ran the full 95-test suite after each refactor; synchronized Celery (beat/queue) and
  Redis (SSE channel) configuration.

## Learnings / technical decisions
- **Async + SSE** is the right shape for long jobs (AI generation): return a task id and
  push the "done" over SSE to keep the API fast and the UX alive.
- The correct tenant key **follows how the system resolves identity**, not what feels
  natural in the domain. I started on the organization key, moved to the user key, and
  landed on a finer-grained key — two migrations that a look at auth first would have
  avoided.

## What I could do better
- The second tenant refactor changed models and services but I **forgot to commit the
  migration** (a follow-up patch fixed it). Schema and ORM changes must ship in the same
  commit, or the schema drifts from the code. A "model + migration + services together"
  checklist prevents it.
- Three tenant keys in two days means I skipped an explicit data-architecture decision at
  the start. A three-line ADR before touching the schema is worth it.
