### 3. Fixing local Docker container errors (Redis port conflict)
- **Symptom:** the Redis container couldn't join the Docker network, leaving **all workers without
  DNS resolution of the `redis` hostname**.
- **Root cause:** the **host's native Redis** (WSL/Linux) already occupied `127.0.0.1:6379`,
  colliding with the container's port mapping.
- **Fix:** change Redis's **external** published port to `6380:6379`. Key design point: inside the
  Docker network the workers still connect via `redis:6379` (the **internal** port, untouched);
  only the port **published to the host** — the one that collided — changes. Closes a tracked issue.

### 5. Security fix for the CRM webhook — closing the last high-severity finding
- The last open high-severity item (noted on the 18th): the CRM webhook didn't verify the sender's
  authenticity.
- **Fix:** a verification function compares the incoming webhook-token header against a configured
  secret using **`hmac.compare_digest`** (constant-time comparison, safe against timing attacks).
  - **Backward compatibility:** with no secret configured → only a log *warning*, no rejection (so
    existing integrations don't break while the secret is rolled out).
  - **With the secret set:** **401** if the token is wrong or missing.
  - `except` ordering: re-raise the `HTTPException` **before** the generic handler so the 401
    propagates instead of being swallowed.
  - **4 unit tests** cover the scenarios: no secret / correct / incorrect / missing token.

  variables**. **Pending:** more **end-to-end tests of the whole integration**.

## Learnings / technical decisions
- **Distinguish internal vs published port.** The container listens on `6379` inside the Docker
  network (workers resolve it as `redis:6379`); the conflict was only on the port **published to
  the host**. Moving the external one to `6380` resolves the collision without touching the
  internal network. Understanding that `host:container` mapping was the key.
- **Use `hmac.compare_digest` to compare secrets**, never `==`: a normal comparison short-circuits
  on the first differing byte and leaks information through timing. The constant-time one doesn't.
- **Safe rollout of a new check:** with no secret configured, *warn* instead of reject; once the
  token is deployed, *enforce* with a 401. This keeps the existing integration from breaking at
  deploy time.
- **Mind the generic `except` that swallows the HTTPException.** Re-raise the `HTTPException`
  before `except Exception`, or the 401 turns into a 500 / gets silenced.
- **"Portable" is proven on another machine.** The Redis conflict only appears on a host that
  already runs Redis; testing on an external PC is what exposes it. Standardizing environments
  requires testing them outside your own.

## What I could do better
- The Redis port conflict was foreseeable: publishing `6379` to the host was always going to clash
  with any native Redis. When exposing a "popular" port, pick an alternate (or don't publish it) from
  the start.
- Security is **closed on high-severity findings**, but the **full integration still needs
  end-to-end testing**. Logged explicitly as next week's follow-up.
