# 2026-06-24 (Wednesday)

**Context:** Backend on a fintech SaaS platform (client) — secrets in CI (end-to-end validation) ·
DB schema bootstrap in the Dockerized dev env · dev↔prod parity (managed-ML credentials, workers) ·
testing the Docker image.

A day to **validate that the Dockerized dev really behaves like production**. Three blocks: (1)
finish moving **all API keys into CI** as independent secrets and confirm the container picks them
up; (2) solve the **DB schema bootstrap** so the Dockerized dev starts with the real schema; (3) test
the image against the product and hunt down an **autocomplete** bug.

## What I did

### 1. Secrets: API keys and config in CI (validation)
- Moved **each sensitive env var into an independent secret**; non-sensitive config into *Variables*.
- **Validation:** confirmed **all API keys are read from CI** at container startup (not from a
  leftover local `.env`). I'm folding the `.env` residue into the compose and re-testing, so there's
  no parallel source of truth.
- **Ordering decision:** if configuring something would require **generating ~200 files**, don't —
  centralize config in **one place**, not one file per variable.

### 2. DB schema bootstrap in the Dockerized dev
- **Reverted** a migration I'd added the day before for "model-defined tables that were never
  migrated": putting them in as a migration dirtied the chain instead of fixing it.
- Full-schema bootstrap: `alembic upgrade` + `create_all` + **reconcile**, so the Dockerized dev comes
  up with the **whole** schema (including tables the model defines but that weren't migrated).
- Made it **portable**: bootstrap from a `schema_baseline.sql` — a **dump of the real schema** —
  instead of depending on migration state. (Also logged: re-baseline the migration chain as the
  source of truth.)

### 3. dev↔prod parity: managed-ML credentials (the autocomplete bug)
- **Symptom:** testing the Docker image against the product, **autocomplete wasn't serving**.
- **Hypothesis:** a mismatch of the managed cloud credentials/services between dev and prod.
- **Fix:** use the **prod service account** for the managed-ML service in the Dockerized dev (in the
  deploy workflow + dev compose). Resolved: autocomplete came back.

### 4. dev→prod worker alignment
- Aligned the Docker workers to prod: a specific **research concurrency** and **no maintenance
  worker** (prod has no such worker; having it in dev created divergence).

### 5. External coordination: registry account upgrade
- Messaged a teammate to **upgrade the container-registry account** and enable **image security
  scanning** (image insights) on the backend image.
- Left open: why the image **wasn't being pushed** to the registry (picked up on the 25th).

## Learnings / technical decisions
- **Don't put things into a migration that aren't migration debt.** The model-defined tables weren't
  a pending schema change: they're tables the model creates and the migration chain didn't reflect.
  The fix isn't a new migration (dirties the chain) but a **bootstrap** (`create_all` + reconcile)
  and, better still, a portable **baseline dump**.
- **The autocomplete bug was environment, not code.** Same binary, different service account: in dev
  the cloud credentials didn't grant access to the managed-ML service. dev↔prod parity includes **the
  credentials**, not just code and ports.
- **Validate that the secret is *consumed*, not just that it *exists*.** Pushing a key to CI isn't
  enough; confirm the container reads it from there and not from a leftover `.env` masking the
  problem.
- **A "too many files" threshold.** If a config implies hundreds of files, the design is wrong:
  centralize in one place, don't explode into one-per-variable.

## What I could do better
- I burned a whole cycle adding and reverting the model-defined-tables migration. Distinguishing
  "table the model creates" vs "schema change to migrate" up front would have gone straight to the
  baseline dump.
- The autocomplete bug cost me because I started hunting it in code. For a failure that "only happens
  in dev", the first suspect should be **credentials/environment**, not logic.
