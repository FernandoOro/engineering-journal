## What I did

### 1. Testing registration / login / password reset → duplicate-email bug
- **Goal:** validate a migration of the transactional email provider — a **new primary with
  the previous provider as fallback** (the send path tries the new one first and falls back
  on a provider error, with an identical API surface).
- **Finding:** it worked, **but** registration, verification-resend, and password-reset
  emails were each being delivered **up to 7 times**. I opened a tracked bug.
- **Diagnosis (from logs):** a single verification email went out 7× with 7 distinct
  provider message ids in ~64s. **Two-part root cause:**
  1. **Trigger:** the success path read attributes that only existed on the *old* provider's
     response shape. Once the new provider became primary, the email had **already been
     sent** and the code then raised an `AttributeError`.
  2. **Amplifier:** the retry decorator retried on **any** exception, ignoring the
     non-retryable set — so the post-send `AttributeError` re-ran the whole task and re-sent
     the email up to the max-retries limit.
- **Fix:**
  - Build the result object **defensively** (`getattr`) so it no longer couples to a specific
    provider's response shape.
  - Make the retry predicate honor the **non-retryable** exceptions: fail fast on
    programming errors; keep retrying only transient ones.
  - Add a greppable dispatch log at the service choke-point to make duplicate delivery
    observable (provider, recipient, message id).
  - **Regression tests:** an `AttributeError` is not retried, and a retryable exception still
    exhausts its retries.
- **Outcome:** bug closed; fix merged across branches.

## Learnings / technical decisions
- **An error after a side effect + a blind retry = a storm.** The email had already been sent
  when the `AttributeError` fired; retrying "just in case" multiplied it. Retries must
  distinguish **transient errors** (retry) from **programming errors** (fail fast) — never
  retry on a bare `Exception`.
- After a provider migration, the success path was still coupled to the old provider's
  response shape. Reading the response with defensive `getattr` decouples the result from any
  specific provider.
- A greppable log at the service choke-point is cheap and makes observable a problem that
  otherwise only shows up in the user's inbox.

## What I could do better
- The bug reached production because the migration lacked a test covering the **full success
  path** with the new provider's response shape. A happy-path test (not just "did it send"
  but "did it post-process the result correctly") would have caught it earlier.
- Secrets are resolved for local only; production still pending. Logged explicitly so it
  doesn't fall through the cracks.
