## Learnings / technical decisions
- **A user with no verification email is a silently lost user.** The "normal" registration sent the
  email; the alternate creation path (seats/clients) didn't — and nobody noticed until a new user
  couldn't get in. **Alternate user-creation paths** need the **same** side-effect guarantee (send
  verification) as the main one.
- **Test *both* flows, not one.** The bug lived exactly in the difference between seat and client;
  validating only one would have missed it. When two flows share a critical step, test both.
- **Dockerizing prod = the same pattern as local/dev, not a new one.** Reusing the DRY compose that
  already works in the other environments is what buys parity; inventing a different layout for prod
  reintroduces divergence.

## What I could do better
- The verification-email send should sit at a **single point** every user-creation path flows through,
  not duplicated per path (and thus forgotten in one). A clear candidate to centralize.
- I hit the token limit with things half-pushed; better to **push the closeable changes first** before
  starting the next big block.
