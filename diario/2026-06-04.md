# 2026-06-04 (Thursday)

**Context:** Backend work on a fintech SaaS platform (client) — collaborative
document chat + admin coupon activation.

> Developed and tested this day; integrated and deployed the following morning.

## What I did

### 1. Shared-document bug: collaborators saw divergent state
- **Diagnosis:** drafts were stored one row per user (`{user_id}_{chat_id}`). A team
  member wrote to their own draft while the owner read theirs, so a questionnaire that
  builds a single shared brand document ended up inconsistent (e.g. 5/54 vs 4/54
  answered on the same chat).
- **Decision:** one chat = one document — the owner's. Remap members to the owner's key
  on both **read and write**.
- **Implementation:** resolve the effective owner from the form's membership, remap
  member identity to the owner on read and write, and skip the ownership check when a
  remap occurs. All lookups fail-soft (log and continue) so a lookup error never breaks
  the flow.

### 2. Correct single-flow conflict handling
- **Decision:** the chat is a single shared flow; once a question is answered, any retry
  must return `409` regardless of status or author.
- **Implementation:** a single query loads all answered indexes and detects the first
  conflict; guards added before write with explicit logging. Non-blocking on DB failure.
- **Tests:** 9/9 unit, including the inverse stale-owner / stale-member cases.

### 3. Fix: the package-level 409 guard blocked the owner itself
- **Diagnosis:** the package guard blocked owners committing their own package, because
  their pending answers already existed and the table has no per-user attribution to
  tell "my pending" from "someone else's committed".
- **Decision:** drop the package-level guard; the per-question guard is sufficient. Kept
  the helper for when per-user attribution is added to the answers table.

### 4. Admin feature: activate an upgrade coupon
- **Decision:** two endpoints to discover and apply upgrade coupons over existing plans
  at no charge, respecting a one-way plan progression.
- **Implementation:** a discovery endpoint listing applicable active upgrade coupons and
  an apply endpoint that delegates to an internal service applying the upgrade at no
  charge. ~1.7k LOC across service, routes, and spec.
- **Tests:** 47 (31 unit + 16 integration), all green.

## Ops
- Local end-to-end testing of the chat flow with two accounts (owner + member) to
  reproduce and confirm convergence; reviewed the test environment configuration for the
  no-charge activation path.

## Learnings / technical decisions
- Collaborative-consistency bugs are usually **data-model** bugs wearing a logic-bug
  costume. A "one row per user" model for a conceptually shared document is the trap.
- Uniqueness guards belong at the level where you actually have the identity to enforce
  them (per question), not where you don't (per package).

## What I could do better
- The "effective owner" logic ended up split across two modules. Centralizing
  "who owns this chat?" in a single helper would stop a future endpoint from diverging
  again.
