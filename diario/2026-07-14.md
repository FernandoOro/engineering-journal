# 2026-07-14 (Tuesday)

**Context:** A cross-origin (CORS) bug blocking local testing against both non-production and
production environments · started investigating networking two personal machines together · three
chat UX fixes once local testing was unblocked.

A day split into two very different halves. The morning was lost diagnosing why I couldn't run
tests against either environment: both non-production and production were rejecting, via CORS,
requests from the local port I needed for testing a full flow. Since deploying to either server
wasn't permitted that week, the fix had to stay local-only, which pushed me to evaluate whether I
could split compute load between my two personal machines to run the full stack locally. The
afternoon, once the CORS blocker was resolved locally, went to three chat UX fixes.

## What I did

### 1. CORS bug — both environments blocked local testing
- Trying to run a full flow against the backend (local, pointed at either environment), requests
  from the local port hosting an auth server used for testing were rejected by CORS.
- Root cause: the allow-list of origins (and a related same-origin-policy middleware) had a fixed
  set of allowed local ports that didn't include the one in use.
- Added the missing local origin to both blocks, applied identically to both environments' copies
  of the file, to avoid drift between them.
- **Couldn't deploy it:** a restriction that week prohibited touching either server. The change
  stayed applied only on my local machine — not committed (see status note).
- Sent the corresponding email notifying the bug and the fix to whoever had deploy permissions.

### 2. Started investigating networking my two personal computers
- Since the CORS fix couldn't be deployed and the servers couldn't be touched, the only way to keep
  testing the full flow was running everything locally. Started investigating putting my two
  computers on the same network to split the full stack's compute load between them — starts
  today, continues tomorrow.

### 3. Three chat UX fixes (afternoon, once local testing was unblocked)
- A "generating" status indicator while a reply streams in (previously no signal that the agent
  was still working during streaming).
- Rendering bot replies as formatted Markdown instead of plain text.
- Routing **every** conversation entry point to the same real backend thread; previously different
  entry points (e.g. a drawer vs. a full screen) opened separate threads for what should be one
  conversation.

## Ops
- Email sent notifying the CORS bug and the locally-applied fix.
- Change applied to a shared config file (both environments) on disk only, no commit or PR yet —
  pending to formalize.
- Started the two-machine networking investigation (no result yet).

## Learnings / technical decisions
- A **testing-infrastructure** bug (CORS blocking the local flow) can block as much as a product
  bug — deserves the same priority of immediate diagnosis.
- Applying the fix to both environments, even though only one is needed locally, avoids the next
  person who deploys running into the same blocker.

## What I could do better
- Should have opened the commit/PR for the CORS fix **the same day**, even though it couldn't be
  merged or deployed yet — leaving it only on my disk means it depends on me not losing track of
  it, and nobody else on the team sees it until I push it.
