## What I did

### 1. Critical bug: making `DONE` a per-task terminal state
- **Symptom:** an infinite re-execution loop; a task already `DONE` got executed again.
- **Root cause:** `DONE` wasn't treated as **terminal**; the canonical writer and a cascade manager
  could reopen/invalidate a completed task, and a "resurrection healer" revived it.
- **Layered fix (ADR + ~146 lines of tests):**
  - **Layer 1:** a state-regression guard in the **canonical writer** (you can't leave `DONE`).
  - **Layer 2:** an execution gate that blocks re-running a `DONE` task.
  - **Layer 3:** a **cross-worker Redis lock** (15-min TTL) so two workers don't race over the same
    terminal task.
  - Neutralized the cascade manager's invalidation of `DONE`.
  - A recovery **carve-out** for the resurrection healer (don't resurrect terminal tasks).
  - **Unit tests** for the terminal guard.
- **Tests + deploy:** deployed to dev with a **worker restart** and validated that a run reaches
  `DONE` and **stays** there (no re-execution).

### 2. dev↔prod homogenization: collapse the `maintenance` queue onto `default`
- **Diagnosis:** in dev the periodic layer (healer/sync/resurrection) was **orphaned**: it routed to a
  `maintenance` queue that **has no consumer** because prod runs no maintenance worker. Result: in dev
  those tasks never ran → divergence from prod.
- **Consistency review:** compared the dev Docker setup against the prod run target to enumerate the
  queue/worker differences.
- **Fix (Option A):** **collapse `maintenance` → `default`** in the Celery config, so the default
  worker (which exists in both) consumes those tasks. Documented in an ADR.
- **Deploy + validation:** deployed to dev with a worker restart and confirmed the **healer runs** and
  an `ALMOST_DONE → DONE` task progresses correctly.
- Logged the remaining debt: the general worker's `-Q` omits queues the local run script lists.

## Learnings / technical decisions
- **A "terminal" state has to be *defended* at every layer.** Writing `DONE` isn't enough: the
  canonical writer, the execution gate, the cascade, and the healer can each reopen it. The defense
  was layered + a **cross-worker Redis lock** for the worker race.
- **A queue with no consumer is work that silently falls through the floor.** Prod has no maintenance
  worker; having a `maintenance` queue in dev made the periodic layer "exist" but **never run**.
  Collapsing it onto `default` (consumed in both) removes the divergence.
- **"Homogenize with prod" means enumerating differences, not intuiting them.** The key step was
  comparing dev against the prod run target and **listing** the queues/workers that differed.
- **One ADR per structural decision.** Writing down the *why* is what stops someone from
  "de-homogenizing" dev three months later.

## What I could do better
- The infinite loop existed because `DONE` was never treated as terminal **by design**. A state
  machine should declare its valid (and forbidden) transitions up front, not get patched with four
  guards after the fact.
- The orphaned queue had been diverging dev from prod for a while with nothing flagging it. I need a
  startup check for "every declared queue has a consumer" that catches it on its own.
