# 2026-06-11 (Thursday)

**Context:** Backend on a fintech SaaS platform (client) — SEO engine, data layer.

## What I did

Kicked off a large feature — an SEO engine (Search Console sync, AI blog generation,
weekly content ideas, and WordPress publishing). It's a **standalone product** inside the
backend (its own routes, like the platform's other products, sharing no code with them),
modeled on an existing in-house SEO product taken as reference, and planned up front with
full specs (architecture, database, services, ER diagram, C4) so implementation leaves no
open questions. The right first step is the data layer: a solid schema and models before
any services or routes sit on top.

### 1. Schema: seven new tables
- **Decision:** model the domain across seven tables (connectors, search-console pages
  and queries, blog posts, idea batches, ideas, and a publish log).
- **Implementation:** an Alembic migration with an explicit down-revision,
  `updated_at` triggers, and **partial indexes** for the hot "opportunity" queries
  (striking-distance and untapped keywords) so the index only covers the rows that matter.

### 2. ORM models
- **Implementation:** five ORM model modules, with the Alembic environment updated to
  import them so autogenerate can see them. ~680 LOC across 8 files.

## Ops
- Ran `alembic upgrade` on the dev database to create the tables and verify triggers and
  partial indexes applied; confirmed the migration environment loaded the new models.

## Learnings / technical decisions
- **Partial indexes** (`WHERE <flag>`) are the right tool for opportunity queries: the
  index stays small and fast because it only covers the relevant rows.
- An explicit down-revision keeps the migration history linear and avoids divergent
  heads — the kind of thing that bites at the worst possible time (a deploy).

## What I could do better
- I picked the multi-tenant key up front without checking how the rest of the system
  resolves identity. That turned out wrong and cost two refactors the next day. A quick
  read of the auth model **before** writing the migration would have saved them.
