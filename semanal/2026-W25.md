# Week 2026-W25 (Jun 16–19)

A **foundations** week on the backend of a fintech SaaS platform (client): production
reliability (a duplicate-email incident), a reproducible local Docker environment to
standardize the whole team, and the **complete closure of the integration's high-severity
security findings**. Fewer new features, more solid groundwork — the kind of work that
isn't flashy in a demo but lets everyone else move faster. Plus a continuous layer of
DevOps/platform work threaded through every day.

## Backend (fintech SaaS platform, client)

- **Shipped (reliability / email):** validated a transactional-email provider migration (a
  new primary with the previous provider as fallback) and **hunted down a duplicate-email
  bug** — registration, verification, and password-reset emails were each being delivered
  **up to 7 times**. Traced it from production logs (one email, 7 distinct provider
  message-ids in ~64s) to a **two-part root cause**: a success path coupled to the *old*
  provider's response shape (so it threw **after** the email was already sent) plus a retry
  decorator that retried on **any** exception (so a post-send programming error re-ran the
  task and re-sent the email). Fixed both with a defensive result-builder, a retry predicate
  that honors the non-retryable set, a greppable dispatch log at the send choke-point, and
  **regression tests**.
- **Shipped (local Docker environment):** a reproducible `docker-compose` stack any teammate
  can bring up identically — Postgres on a vector-capable image (semantic-memory tables need
  the `vector` extension the base image lacks), the Celery monitoring UI fixed so it installs
  in the container, **healthchecks** so Compose knows when services are truly ready, database
  **credentials moved out of the versioned YAML** into an env file (security fix), exposed DB
  ports for external access, **Redis persistence** (named volume + RDB snapshotting), and a
  fix for a **host port conflict** (the host's native Redis on 6379 blocked the container from
  joining the network and left all workers without name resolution — republished on an
  alternate host port while keeping the internal port untouched).
- **Shipped (integration security — high-severity list cleared):** ran a sweep over a
  third-party CRM/accounting integration and **closed five findings** (catalog grew 24 → 28
  documented). Enforced **authentication at the router level** on routers that had none
  (~25 CRM endpoints plus infrastructure controls), raised infrastructure operations to
  **admin-only**, fixed a **broken OAuth callback** that always ran a legacy stub (verified
  live), **mounted an admin router that was never wired in** (so its endpoints were
  unreachable), and added **webhook sender verification** using a constant-time token
  comparison with a safe rollout (warn when unconfigured, enforce a 401 once the secret is in
  place) and four unit tests. This cleared the integration's **entire high-severity backlog**.
- **Shipped (secrets in CI — groundwork):** a first CI workflow on a self-hosted runner,
  starting the migration of secrets out of `.env` files into managed CI secrets — each `.env`
  shuttled by hand is a leak surface.
- **Technical decisions:**
  - Constant-time comparison for secrets, never plain equality (which leaks via timing).
  - **Defensive auth at the router** (`dependencies=[Depends(...)]`) instead of trusting each
    handler to remember it.
  - Privilege **by operation** (infrastructure controls → admin role), not merely "logged in".
  - Safe rollout of a new check: warn without the secret, enforce once it's present.
  - **Internal vs published port**: move the external port without touching the internal
    network the workers resolve by name.
  - Persistence via a host-backed named volume (and a snapshotting policy for the cache).
  - Reading a provider's response defensively so the success path doesn't couple to any one
    provider's shape.
- **Mistakes (root cause):**
  - The duplicate-email bug reached production because the provider migration **lacked a
    full success-path test** — only "did it send?" was covered, not "did it post-process the
    result correctly?".
  - A credential misalignment took two commits because I didn't define a **single source of
    truth** for the env file up front.
  - The host port conflict was foreseeable: publishing a "popular" port to the host was always
    going to clash with a native service.
- **What I learned:** an error **after a side effect** (the email was already sent) plus a
  blind retry equals a storm — retries must distinguish transient failures from programming
  errors. And "portable" is only proven **on another machine**: testing the bring-up on an
  external PC is what surfaced the port conflict.
- **Where I'm weak:** full happy-path coverage when integrating a provider (not just the
  send), and anticipating environment conflicts (popular ports) before exposing them.
- **Focus next week:** **end-to-end QA of the whole integration**; resume secrets-in-CI for
  dev/prod once the required admin tokens are provided; close the final low-priority security
  item (an owner-action).

## Operations / DevOps (cross-cutting, every day)

Platform work that underpins everything above. It doesn't always leave a commit, but it's part
of every workday and a real share of the effort:

- **Deployments to dev and prod:** pushed changes to whichever environment a task needed —
  validating the email provider and the SEO engine PoC **in production**, and deploying +
  administering the **dev server** to be able to test the CRM webhook end to end.
- **Server administration:** monitoring and control of the production server, management of the
  dev server, and direct work on the servers themselves (some changes are committed straight
  from the production host).
- **Per-environment `.env` management:** creating and adjusting environment variables as needs
  arose — a webhook token, database credentials moved into an env file to get them out of the
  versioned YAML, and the local-stack variables — plus **syncing `.env` across servers** so a
  change in one environment doesn't break another.
- **Secrets in CI:** first step toward pulling `.env` files out of the manual flow and into
  managed CI secrets (a workflow on a self-hosted runner).
- **Container lifecycle:** build / bring-up / restart of the Docker stack with healthcheck
  verification — tests take real time while containers restart and become healthy again — plus
  a full bring-up test on an **external machine** to confirm portability.
- **Realistic test data:** a script that clones production data into a local database and seeds
  the new product, so features can be exercised against realistic data without touching prod.
- **Recurring delivery:** branch merges (dev ↔ main), propagating scripts across branches, and
  resolving post-deployment issues.

## Global retrospective
- **Pattern of the week:** a healthy pivot from "ship features" to "harden the foundations".
  The infrastructure work (a reproducible Docker stack, secrets, CI) and clearing the entire
  high-severity security backlog compound the team's future velocity, even if they don't produce
  an eye-catching demo.
- **Habit to reinforce:** a **full success-path test** as part of any provider/integration
  change, and an **integration checklist** (does the router have auth? does the real callback
  run? is it mounted in the entrypoint? is the port going to conflict?) that would have caught
  all the integration findings and the port conflict up front.
- **External blocker:** the secrets-in-CI migration depends on tokens only the repo admins hold
  — ask for them at the start, not when hitting the wall.
- **Logged debt (accumulated):** move secrets to **production** (local already done); end-to-end
  QA of the integration; rotate credentials still present in git history (from W24).
