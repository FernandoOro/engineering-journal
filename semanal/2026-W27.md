# Week 2026-W27 (Tue Jun 30 – Fri Jul 3)

A week of **finishing the platform + starting a method**. I closed out the Dockerization of all three
environments (prod now runs self-contained and closed to the outside), fixed a reliability bug that
left new users unable to verify, and started a new **objective-based, spec-driven methodology**
(Specify → Plan → Tasks → Validate, *Done* only with evidence) with the **Phase 0** work assigned to
me: fix an orphaned task queue, add a per-user run-rate limit, and — most importantly —
**instrument token usage at the LLM boundary**. That telemetry ended up **surfacing a silent
production failure** (the managed LLM was down; everything was falling through to a secondary
provider). A foundations week with a lot of **testing** and **server operations**.

## Backend (fintech SaaS platform — client)

- **Done (auth reliability):** users created via **seat/client** (an alternate creation path) **never
  received the verification email** → locked out. Fixed + tested **both flows** and merged to main.
- **Done (Dockerization + prod network hardening):** consolidated the prod compose into a **single
  file** (removed a duplicate); Redis+Postgres as self-contained containers; **prod exposes no ports**,
  the monitoring UI bound to localhost; dev **does** expose DB/Redis for inspection — a deliberate
  asymmetry.
- **Done (fallback email provider):** migrated the fallback provider and tested it in dev; its API key
  is pending configuration by management.
- **Done (schema/DB):** removed scaffold tables + created a missing one; backed up prod data into the
  Docker DB container; progress toward a single migration branch.
- **Done (Phase 0 — my assignment):**
  - **Orphaned task queue:** 8 scheduled entries pointed at a queue with no worker → a periodic sync
    never ran. Fixed by defining a dedicated queue + worker (isolated from the default queue).
  - **Per-user/day run-rate limit** on the streaming endpoint — an atomic Redis counter
    (`INCR`+`EXPIRE`), a **typed 429**, a fail-open feature flag. **~132 lines of tests** + integration
    fixtures.
  - **Usage at the LLM boundary ★ root unlock:** the boundary returned a bare string, discarding
    `usage`. Now returns **text + a usage struct** normalized per provider (with a compatibility shim).
    Without it there was no cost/budgets/metrics anywhere in the product.
  - **Token/cost on the agent result:** usage/cost fields emitted through the single `execute()` choke
    point + topology assembly + **per-run persistence**. **~493 lines of tests** (agent-usage
    telemetry + run topology).
- **Done (deploy fix):** a **single runner** served both dev **and** prod → a merge to main took down
  dev. Fixed with **dedicated runner labels** per environment. Along the way, reverted a blind
  `--force-recreate`.
- **Done (bugs surfaced while testing, day 3):** the 429 body had to be a plain string, not a nested
  object; `await` the persistence directly instead of fire-and-forget; map the secondary provider's
  different usage shape (otherwise token counts were null).
- **Done (root finding):** persisting `model/provider` showed **everything** was going through the
  secondary provider, never the managed one. Cause: the cloud **service-account secret contains
  invalid JSON** → it blocks the managed LLM and **silently forces fallback**. Opened an issue + a
  discovery spec.
- **Technical decisions:**
  - **Prod exposes zero data ports; dev may open them** (network-surface asymmetry).
  - **One deploy file per environment**, **one runner per environment** (dev/prod decoupled).
  - **Capture usage at the boundary** = instrument everything through a single seam (Template-Method).
  - **Atomic rate limit + fail-open**: if it isn't atomic it doesn't work; and a new gate must be
    disableable without a redeploy.
  - **Direct `await` for writes that matter** (not fire-and-forget).
  - **A queue with a worker, or route to the default — never half-done** (same pattern as last week's
    orphaned queue).
  - **Spec-driven: Done only with real production evidence.**
- **Mistakes (root cause):**
  - The verification email was **duplicated per creation path**, so it was missing on the alternate
    one. It should be centralized in a single point.
  - The `--force-recreate` was blind against stale mounts; I should have diagnosed which mount was
    stale first.
  - I split the boundary/telemetry work across two days due to a **token limit** — again, from not
    slicing into small PRs from the start (exactly what the method asks).
- **What I learned:** **telemetry pays for itself** — instrumenting `usage` wasn't only for budgets;
  it's what made **visible** that the primary provider had been down for a while. What you don't
  measure, you don't see; a fallback that fires **every time** should **alert**, not just log.
- **Where I'm weak:** slicing big work into small PRs (a token limit keeps splitting root-unlock tasks
  when I don't); and proactive alarms for silent failures (perpetual fallback, a primary that never
  serves).
- **Next-week focus:** resolve the invalid service-account secret (coordinate with management) + add a
  **perpetual-fallback alert**; continue Phase 0 (stretch: per-attempt timeouts / health-gating the
  model carousel); finish the backup restore into the Dockerized DB.
