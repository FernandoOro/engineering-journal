# Week 2026-W29 (Mon Jul 13 – Fri Jul 17)

A week of **closing out a client SDK rewrite + a new project-management surface**, with two days
(Tuesday-Wednesday) eaten by a testing-infrastructure blocker: a CORS bug preventing any local
testing against either environment, with no permission to deploy a fix to either server. The real
unblock wasn't more compute (a failed attempt at networking two personal machines) but **running
less**: bringing up only the containers actually needed instead of the full application. From there
on, three solid days wiring real data into a chat feature and five new project-management screens.

## Frontend / client SDK (fintech SaaS platform — client)

- **Done (Monday — closing out the SDK rewrite):** merged three stacked branches carrying a full
  SDK rewrite: the client rebuilt as a single unified client (handles a 7-event streaming protocol),
  UI components wired onto the new client, an edge-case empty-response status handled, a round of
  code-review fixes (fetch gating, abort handling, ids, selectors), aborted turns that used to get
  stuck pending now resolve correctly, and the old dead streaming-consumer implementation deleted.
  Reorganized each piece into its correct directory and validated with the full unit + integration
  suite before merging to main — no new code that day, since it was already written; the work was
  integration and verification.
- **Done (Tuesday — chat UX, once local testing was unblocked):** a "generating" status indicator
  during streaming, Markdown rendering of bot replies, and routing **every** conversation entry
  point to the same real backend thread — previously different entry points opened separate
  threads.
- **Done (Wednesday):** a drawer-based chat surface brought to parity with the main chat screen —
  two surfaces that had diverged.
- **Done (Thursday — wiring real data):** live data wired into 4 surfaces that previously used mock
  data: a project switcher, a project-management screen, 3 stat cards on an overview screen, and
  team/member cards.
- **Done (Friday — SDK reorganization + billing UI):** moved calls to the SDK modules matching their
  actual business domain instead of where they were first written; a project-creation modal wired to
  a plan-creation endpoint (billing cycle + coupon + name); 4 billing UI fixes — show the coupon's
  discount after applying it, hide the billing-cycle toggle for fixed-price plans and show the
  discounted price, the top-tier plan hides the coupon and changes its submit button to "Contact
  us", properly separate two fields that had been unintentionally linked; and a fix for a project
  switcher that looked like it worked but didn't actually change the active project.
- **Technical decisions:**
  - Rewrote the client SDK instead of patching the old one — it didn't support the new streaming
    protocol; patching would have been debt on top of debt.
  - Organized SDK calls **by business domain** instead of by feature — so each call lives where its
    domain logic actually belongs, not where it was first used.
  - A single real conversation thread for all entry points — avoids divergent state between chat
    surfaces.
- **Mistakes (root cause):** two chat surfaces diverged because a new one (drawer) was added without
  syncing changes already made to the existing one — missing a single source of truth for chat
  behavior. A project switcher "looked like" it worked but didn't change the active project — a
  live-data feature was closed without verifying the end-to-end flow with a real click.
- **What I learned:** when two surfaces render the same domain (chat in a drawer and in a full
  screen), treat them as one component with two shells, not two implementations — otherwise every
  fix has to be applied twice, and one always gets missed.
- **Where I'm weak:** end-to-end verification of UI flows before closing a ticket — the switcher bug
  would have been caught with a single click before marking the feature done.
- **Next-week focus:** finish moving any SDK call still in the wrong module; manual QA of the three
  chat surfaces together (drawer, full screen, generating status) to avoid parity regressions.

## Backend (fintech SaaS platform — client)

- **Done (Tuesday — CORS bug blocking testing):** both non-production and production rejected, via
  CORS, requests from the local port used for testing. Added the missing origin to both blocks in
  the shared server config — fix applied but **not deployed** (no deploy permission that week), so
  it stayed **local-only** (change not committed). Sent the corresponding email notifying the bug.
- **Done (Wednesday-Thursday — a new project-management surface):** 5 new endpoints: a project list,
  a project-management view (status/plan/progress), a project-overview view (plan, progress,
  product usage), a team view, and a paginated members view.
- **Done (Thursday — data fix):** a plan-name field was being resolved via an unnecessary join —
  removed, since the field was already available directly.
- **Done (Friday):** plan creation now accepts an optional project name, to associate the plan with
  a project at creation time.
- **Technical decisions:**
  - Applied the CORS fix to **both** environments (same shared file) even though only one was
    needed locally — avoids the next deploy running into the same blocker.
  - Kept the plan-name field as a direct alias with no join — the data was already denormalized;
    the join was complexity without real benefit.
- **Mistakes (root cause):** the join assumed a relationship that didn't reflect the real model —
  accidental complexity added without checking the schema first. The CORS fix **stayed uncommitted
  with no PR open for an entire week** because of the deploy restriction — a real risk of losing it
  if not prioritized.
- **What I learned:** before building a new endpoint on top of a field, it's worth checking what's
  already denormalized in the model — avoids unnecessary joins like this one.
- **Where I'm weak:** left a real blocking fix (CORS) without a commit or open PR for a full week. I
  should have opened the PR anyway even if it couldn't be merged or deployed, instead of depending
  on it staying only on my disk, and to give the team visibility.
- **Next-week focus:** commit and open the PR for the CORS fix (deploy still pending approval, but
  registered); check whether other endpoints repeat the unnecessary-join pattern.

## Operations / local testing (cross-cutting, Tuesday-Wednesday)

Almost two full days resolved by testing infrastructure, not product work:

- **Networking my two personal computers:** tried configuring them on the same network to split
  compute load for running the full application locally. Spent a good part of Tuesday and Wednesday
  on this — **didn't work**; the real bottleneck was local compute power, not networking.
- **Partial application execution (this worked):** instead of bringing up the full app, investigated
  and got running **only the basic containers** needed for the code being written. This replaced the
  need for the two-machine network and was what actually unblocked testing from Wednesday on.

## Global retrospective

- **Pattern of the week:** two days resolved by a testing-infrastructure limitation (CORS + the need
  to run locally) rather than by the product work itself — and the real unblock wasn't more compute
  (the two-machine network) but **running less** (partial containers). Same underlying pattern as
  previous weeks: the simple solution beats the one that sounds more powerful.
- **Habit to change:** when a fix blocks the team's work but can't be deployed (CORS), open the
  commit/PR anyway instead of leaving it floating unrecorded — it shouldn't depend on memory or on a
  single machine's local environment.
- **Logged debt:** the CORS fix, still uncommitted (both environments); check other endpoints for
  the same unnecessary-join pattern.
